# Raft 过程详解与必要性分析

这份文档不假设你读过 Raft 论文。我们从"如果不要 Raft 会怎样"出发,一步步推出 Raft 每个机制的必要性。读完后你应该能回答:为什么有 Leader、为什么要多数派、为什么有日志、为什么要快照、为什么读可以不过 Raft。

---

## 第一部分:先理解我们要解决的到底是什么问题

我们的运维平台要把"服务器清单"这样的状态存起来。最朴素的方案是单机内存:

```
一台机器, 内存里一个 map, 直接读写。
```

这个方案有三个致命问题,Raft 的每一个机制都是为了解决其中之一或多:

1. **可靠性**:这台机器挂了,数据丢失,整个平台瘫痪。
2. **一致性**:如果为了可靠做多副本,两台机器同时收到不同的写,数据会分叉。
3. **可用性**:单机挂了就不可用;但多副本又带来一致性问题,如何在不牺牲一致性的前提下提高可用性?

Raft 的答案:多副本 + 强一致 + 容错。它的核心承诺是:**只要多数节点活着,数据就不丢、不冲突,且系统可用**。

---

## 第二部分:为什么要有 Leader(而不是所有节点都能写)

### 假设方案:多主(任何节点都能接写)

三台机器 A/B/C,都直接接写。两个客户端同时操作:
- 客户端1 向 A 写"x=1"
- 客户端2 向 B 写"x=2"

A 要把 x=1 同步给 B/C,B 要把 x=2 同步给 A/C。最终三台机器上 x 的值是什么?**无法确定**。这就是"多主冲突",需要复杂的冲突解决算法(向量时钟、CRDT 等),且无法保证强一致。

### Raft 的方案:单 Leader

任何时刻只有**一个节点是 Leader**,所有写必须经过它。Leader 决定日志顺序,然后把同样的顺序复制给所有 Follower。

```
客户端 ──写──> Leader ──复制──> Follower1, Follower2
        读也优先找 Leader
```

### 为什么必须这样

- **确定的全局顺序**:Leader 收到的写,按到达顺序编号(日志索引 index),所有节点按相同 index 顺序应用,结果必然一致。这是消除冲突的根本。
- **简化一致性**:不用解决冲突,因为没有冲突源——只有一个写入口。
- **为什么不是"只读"也找 Leader**:后面"读"那节会讲,Leader 读能保证读到最新已提交数据。

### 代价

Leader 是单点。如果 Leader 挂了,必须选新的。选举期间(通常几秒)不可写。这是为了强一致付出的可用性代价,被多数派机制控制在可接受范围(见下节)。

## 第三部分:为什么是多数派(Quorum),不是全部或一个

Leader 要把一条日志复制给 Follower。复制到几个算"安全"?

### 选项1:复制到全部节点才算成功

三台 A(Leader)/B/C。Leader 写一条日志,要等 B 和 C 都确认。

问题:**任何一个 Follower 慢一点或网络抖动,写就卡住**。3 节点里只要 1 台慢,整个系统写性能被最慢节点拖垮。这是"木桶效应"。

更糟的是:C 挂了,系统完全不能写——因为永远凑不齐"全部"。这比单机还脆弱(单机挂了只是不可用,这里是主动放弃可用性)。

### 选项2:复制到 1 个(Leader 自己)就算成功

Leader 写完本地就返回成功,不复制。

问题:**Leader 一挂,这条日志可能还没来得及复制出去,数据丢失**。等于没有多副本,可靠性 = 单机。

### Raft 的方案:多数派(N/2 + 1)

3 节点集群,Leader 把日志复制给**至少 2 个节点(含自己)就算"已提交(committed)"**。即 Leader 自己 + 1 个 Follower 确认,就可以安全返回客户端成功。

### 为什么多数派是"恰好"的答案

这是 Raft 最精妙的数学性质,理解它你就理解了 Raft 的灵魂:

**任意两个多数派,必然有交集(至少一个公共节点)。**

- 3 节点的多数派是 2。任意两个"2 节点子集"在 3 节点里必有交集。
- 5 节点的多数派是 3。任意两个"3 节点子集"在 5 节点里必有交集。

这个交集性质保证了:**任何已提交的日志,在新 Leader 选举时必然存在于多数派节点上,因此新 Leader 一定拥有所有已提交日志**。

### 反证:如果新 Leader 可能没有已提交日志会怎样

假设 3 节点 A/B/C,A 是 Leader。一条日志只复制到 A 自己(没复制到 B/C),A 就返回客户端"成功"了——这违反多数派原则,但假设我们允许。然后 A 挂了,B 当选新 Leader。B 上没有这条日志,数据丢失,而客户端已经收到"成功"。**这就是灾难**。

多数派原则堵死这条路径:已提交 = 至少在多数派节点上;新 Leader 必须来自多数派(选举也要多数派投票),所以新 Leader 一定见过所有已提交日志。

### 容错能力

- 3 节点(多数派 2):容忍挂 1 台。挂 1 台还能凑 2 个,继续写。
- 5 节点(多数派 3):容忍挂 2 台。
- 一般不用偶数节点:2 节点多数派还是 2,容忍挂 0 台(挂 1 台就凑不齐),比单机没强多少还更复杂。

## 第四部分:为什么有日志(Log),而不是直接改状态

Raft 维护一个**只追加(append-only)的日志序列**,每条日志是一条命令。状态(FSM 的 map)是"应用日志后得到的结果"。

```
日志:  [cmd1: add s1] [cmd2: add s2] [cmd3: ...]
                  │           │
状态(map):  应用cmd1后 {s1}  应用cmd2后 {s1,s2}
```

### 为什么不直接改 map

假设直接改 map,没有日志。Leader 挂在"改了本地 map、还没来得及复制给 Follower"之间。新 Leader 不知道 Leader 改过什么,数据丢失且无法恢复——因为"改 map"这个动作没有留下可追溯的痕迹。

### 日志的作用:可追溯、可重放、可复制

1. **可追溯**:每条写都是一个日志条目,有唯一递增的 index。Leader 挂了,新 Leader 能看到"到 index 5 为止的日志",精确知道已经做了什么。
2. **可重放**:节点重启后,从快照恢复基础状态,然后按 index 顺序重放快照之后的日志,重建出当前状态。这就是为什么我们 FSM 用内存 map,重启数据还在——Raft 重放了日志。
3. **可复制**:Leader 把日志条目按 index 发给 Follower,Follower 按相同顺序 append,保证所有节点的日志序列一致。复制"一段有序字节"比复制"一个 map 的最终状态"简单且可靠。

### 日志与状态的关系:事件溯源(Event Sourcing)

这其实就是事件溯源模式:**日志是真相(source of truth),状态是日志的投影**。数据库的 redo log、Kafka 的 stream、比特币的区块链,本质都是这个思路。Raft 把它用在了强一致共识上。

### 关键概念:committed(已提交)

一条日志"committed"意味着它已被复制到多数派节点,**此时它被认为安全,不会丢**。Leader 只在日志 committed 后才返回客户端成功。未 committed 的日志(Leader 挂在复制途中)可能丢失,但客户端还没收到成功,所以这是安全的"失败重试"语义。

## 第五部分:选举(Election)——Leader 挂了怎么办

### 触发:Follower 超时未收到心跳

Leader 定期向所有 Follower 发心跳(空 AppendEntries RPC),告诉它们"我还活着"。Follower 启动一个**选举超时计时器**(随机 150-300ms),如果超时没收到心跳,就认为 Leader 挂了,发起选举。

### 选举超时为什么要随机化

如果所有 Follower 同时超时,会同时变成 Candidate 同时拉票,可能谁都拿不到多数票(票数分散),导致选举失败重试,循环往复。随机化超时让某个 Follower 先超时、先拉票,大幅降低冲突概率。这是 Raft 工程上很关键的一个细节。

### 过程

1. Follower 自增 term(任期号),变成 Candidate,先投自己一票。
2. Candidate 向所有节点发 RequestVote RPC:"选我,我的日志到 index N、term T"。
3. 其他节点收到后判断:
   - **候选人的日志是否至少和我一样新**(比较最后一条日志的 term 和 index)。如果候选人日志更旧,拒绝投票。**这条规则保证新 Leader 一定有所有已提交日志**(配合多数派)。
   - 本 term 内还没投过票,就投给候选人。
4. Candidate 收到**多数票**就当选新 Leader,立即发心跳宣告。

### 为什么候选人日志必须"至少和我一样新"才能拿到票

这是 Raft 的"Leader 完整性"保证。已提交日志必然在多数派节点上。如果允许日志更旧的节点当选,它可能缺少已提交日志,导致已提交数据丢失。投票规则"只投给日志至少和我一样新的候选人"加上"选举需要多数派",数学上保证新 Leader 包含所有已提交日志。

### term(任期)的作用

term 是单调递增的整数,每次选举自增。它像"朝代号",防止旧 Leader 复活搞乱。如果一个节点的 term 比别人低,它的任何消息(投票、日志复制)都会被忽略。这避免了"网络分区恢复后,旧 Leader 还在写"的脑裂问题。

## 第六部分:脑裂(Split-Brain)与 Raft 如何防止它

### 什么是脑裂

3 节点集群,网络把 A(Leader) 和 B 分在一区,C 单独在另一区。C 收不到心跳,超时后自己发起选举——但它只有 1 票,凑不齐多数(需要 2),无法当选。所以 C 那一区**不能写**,只能等网络恢复。

同时 A+B 区有 2 个节点,是多数派,可以正常选主/写。

### 为什么不会出现两个 Leader 同时写

假设极端情况:A+B 区 A 当 Leader 在写,C 区如果(错误地)也当了 Leader 在写——两个 Leader 各写各的,数据冲突。这就是脑裂。

Raft 用**多数派**彻底堵死这条路:
- 任何写必须复制到多数派(2 票)才 committed。
- 任何选举必须拿到多数票(2 票)才能当选。
- 两个区不可能同时各有 2 票(2+2=4 > 3)。

所以**网络分区时,少数派区必然不能写也不能当选**,只有多数派区能继续工作。少数派区的写请求会失败(或排队等待),数据一致性得以保全。这就是"在 CAP 中选了 CP"的含义——分区时牺牲可用性保一致性。

## 第七部分:快照(Snapshot)——为什么不能只靠日志

### 日志无限增长的代价

每条写都是一条日志。系统跑一年,日志可能有几百万条。问题:

1. **重启慢**:重启要重放所有日志重建状态,几百万条要很久。
2. **磁盘占满**:日志文件无限增长。
3. **新节点同步慢**:新节点加入,要从第一条日志开始重放,太慢。

### 快照的方案

定期把**当前状态(map 的内容)**完整打包存下来,然后**丢弃快照点之前的旧日志**。重启时:先加载快照(瞬间恢复到快照时刻的状态),再重放快照之后的少量日志。

```
旧日志: [1][2][3][4][5][6][7]   <- 这些可以丢
快照@5: {s1, s2}                  <- 状态到 index 5
新日志: [6][7]                    <- 只需重放这两个
```

### 必要性:这是工程必需,不是算法必需

注意:理论上光靠日志也能正确工作(无限重放)。快照是为了**性能和可用性**,不是正确性。但生产环境没有快照不可行——重启时间和磁盘占用会变成灾难。所以快照是 Raft 实现的标配。

### 我们的代码对应

- `FSM.Snapshot()`:打包当前 map 成快照。
- `FSM.Restore()`:从快照恢复 map。
- `fsmSnapshot.Persist()`:把快照写到 Raft 的 sink(落盘)。
- `config.SnapshotThreshold = 2`:积累 2 条日志就可能触发快照(演示用的小值,生产环境调大,比如 10000)。
- `config.SnapshotInterval = 30s`:最多 30 秒做一次快照。

这就是为什么我们重启后端,日志里看到"FSM 从快照恢复: 2 台服务器"——它没重放历史日志,而是直接加载了快照。

## 第八部分:Apply 闭环——一条写命令的完整旅程

这是 Raft 运行时的核心循环。以"新增服务器 s4"为例:

```
1. 客户端 POST /api/servers {id:s4,...}
2. api 层: json.Decode → srv
3. store.AddServer:
   a. 构造 command{Op:"add", Server:srv}
   b. json.Marshal → 字节 data
   c. raft.Apply(data, 5s)  ← 提交给 Raft, 返回 future
4. Raft 内部(future 阻塞等待):
   a. Leader 把 data 追加到本地日志 (index=N)
   b. 并行 AppendEntries RPC 复制给所有 Follower
   c. 收到多数派确认(含自己) → 标记 index=N 为 committed
   d. 通知 FSM.Apply 应用这条日志
5. FSM.Apply(l):
   a. json.Unmarshal(l.Data) → cmd
   b. 加写锁 → f.servers[cmd.Server.ID] = cmd.Server → 解锁
   c. 返回 nil(成功)
6. future.Error() 返回 nil → store.AddServer 返回 nil
7. api 层: 201 Created + 返回 srv JSON
```

### 每一步的必要性

- **步骤4a 追加本地日志先**:保证即使复制途中 Leader 挂了,这条日志在日志序列里有位置,新 Leader 能判断它是否存在。
- **步骤4b 并行复制**:降低延迟,不必等最慢节点(多数派确认即可)。
- **步骤4c 多数派才 committed**:保证 committed 的日志不会丢(数学保证,见第三部分)。
- **步骤4d 在 committed 后才 Apply**:FSM 只应用已提交日志,避免应用了未提交日志后 Leader 挂掉导致回滚(我们的 FSM 没有回滚机制,所以必须只在 committed 后 Apply)。
- **步骤5 Apply 的幂等性**:同一日志可能被 Apply 多次(比如日志重放),Apply 必须幂等。我们的 `f.servers[id] = srv` 天然幂等——重复写同一个 key 结果不变。

### 单节点 vs 多节点的差异

单节点:步骤4b 复制跳过(没有 Follower),步骤4c 多数派=自己 1 票即 committed。其余完全一样。这就是为什么我们单节点代码和多节点代码几乎一致——Raft 的逻辑不因节点数变化,只是"多数派"的数学结果不同。

## 第九部分:读为什么不必须过 Raft

我们的 `ListServers` 直接读 FSM 内存,不走 `raft.Apply`。

### 强一致读 vs 脏读

- **强一致读**:经过 Leader 的 read index,保证读到最新已提交数据。多节点场景下,如果允许从 Follower 读,可能读到旧数据(Follower 日志落后)。
- **脏读(我们当前的做法)**:直接读本地 FSM 内存。单节点下没有"旧"的问题,完全正确。

### 多节点时为什么要小心

3 节点,A 是 Leader。客户端连到 B(Follower)读,B 可能还没收到最新的几条日志,读到的是几秒前的状态。如果业务要求强一致读,必须:
- 让读请求转发到 Leader,或
- Follower 用 read index 向 Leader 确认"当前 commit index",再读。

这是 v0.2 多节点之后要处理的问题。单节点和"读旧一点无所谓"的场景,直接读内存是合理优化。

## 第十一部分:一句话总结每个机制的存在理由

- **Leader**:消除多主写冲突,建立单一写入口和全局日志顺序。
- **多数派**:在可靠性与可用性间取得平衡,数学保证已提交日志不丢、新 Leader 完整。
- **日志**:可追溯、可重放、可复制,把"改状态"变成"追加事件",容错的基础。
- **term**:防止旧 Leader 复活搞乱,标识"合法朝代"。
- **随机选举超时**:避免多个 Follower 同时竞选导致活锁。
- **投票时校验候选人日志**:保证新 Leader 含所有已提交日志(Leader 完整性)。
- **快照**:防止日志无限增长,加速重启和新节点同步(性能必需)。
- **Apply 在 committed 后**:避免应用未提交日志后无法回滚。
- **Apply 幂等**:容忍日志重放(重启/快照恢复后),重复应用结果不变。
- **心跳**:Follower 判断 Leader 存活的机制,触发选举的源头。

理解了"每个机制解决什么问题、如果去掉会出什么问题",你就真正理解了 Raft,而不只是记住了流程。建议你回到后端代码,对照这份文档把每个 Raft 概念在代码里找出来,印象会非常深。
