# 架构设计

本文档是 deepsea-ops 项目的总体架构说明, 描述系统目标、拓扑选型、技术栈和演进路径。分布式一致性原理见 [Raft原理详解.md](./Raft原理详解.md), 代码层面见 [后端代码导读.md](./后端代码导读.md)。

---

## 一、项目目标

deepsea-ops 是一个分布式服务器运维平台, 用于管理维护 20+ 台服务器上的:

- Java 微服务 / Java 程序 / Python 程序
- Redis、PostgreSQL、Kafka、Elasticsearch、ClickHouse、Nacos

首期聚焦 Java 微服务管理, 包括:

- 配置文件维护修改: 连接 Nacos、Java 本地配置文件、jar 包内部配置文件进行读取比对
- 扩容与迁移: 对 Java 程序在不同服务器上进行扩容和迁移

整体要求:

- 分布式部署, 20+ 台服务器任意一台都可访问
- 节点间心跳连接
- 一台部署后导入服务器名单, 自动连接、自动部署本项目

本项目作为开源项目维护。

## 二、开源项目参考

调研过的同类项目及其对本项目的借鉴价值:

| 项目 | 架构特点 | 借鉴点 |
|------|---------|--------|
| Jpom (dromara) | Server + Agent 双端, Java 后端 + Vue 前端, Agent 长连 Server | 头号参考, 节点分发、项目管理模式 |
| Spug | Agentless SSH, Python(Django) + Vue | 轻量, 但不满足对等集群诉求 |
| 1Panel | Go + Vue, 单机面板 | UI 与资源监控设计 |

| Ansible / AWX | 无 Agent, playbook 幂等 | bootstrap 自动部署思路 |
| Consul / Nomad | Go, gossip 心跳 + Raft 调度 | 工业级对等集群参考 |
| Apollo / Nacos | 配置中心 | 客户端缓存 + 版本对比 |

业界两条主线: Agent 长连(Jpom 式)与 Agentless SSH(Spug 式)。本项目需求含"任意节点可访问、互相心跳、自动部署", 选 **Agent + 集群** 路线。

## 三、分布式拓扑选型

服务器间网络全互通。对比三种拓扑:

### 模式 A: Server + Agent 主从

- 一个 Server 集群接所有 Agent, Agent 单向长连
- 任意节点访问靠反代硬凑, 非真对等
- 实现最简, 但 Server 单点风险, "任意节点当入口"是伪命题

### 模式 B: 全对等 mesh

- 每台都跑相同节点, gossip 互相心跳
- 真任意节点入口, 无单点
- 强一致写(配置管理)需叠 Raft, 复杂度暴增, 20 台规模过度

### 模式 C: 混合(控制面小集群 + 全员 Agent)

- 3 节点 Raft 控制面 + 所有机器跑 Agent
- Raft 原生强一致, 秒级故障切换
- 入口代理让任意节点 IP 都能访问 UI
- 复杂度介于 A 与 B 之间, 正好匹配 20 台规模

### 选定: 模式 C

选 C 的三个理由:

1. "任意一台可访问、互相心跳" 是对等入口诉求, A 做不到真任意节点, B 代价大, C 用轻量入口代理精准命中
2. "配置文件比对修改" 是强一致写场景, Raft 比 mesh 最终一致靠谱
3. 20 台规模下 B 的 gossip 复杂度是负收益, C 的 Raft 一次性投入后续受益

## 四、模式 C 架构总览

```
┌─────────────────────────────────────────────────┐
│              控制面 (3 节点 Raft)                │
│   N1: Leader   N2: Follower   N3: Follower      │
│   每节点内: API + Raft 引擎 + 调度器            │
│   强一致状态: 服务器清单/服务定义/配置基准/部署计划 │
└─────────────────────────────────────────────────┘
        ▲心跳上报            ▲心跳上报
   ┌─────┴─────┐       ┌─────┴─────┐  ... 共 20 台
   │  Agent@N4 │       │  Agent@N5 │
   │ 执行部署/ │       │ 执行部署/ │
   │ 读配置/   │       │ 读配置/   │
   │ 起停进程  │       │ 起停进程  │
   └───────────┘       └───────────┘
   入口代理(每台都跑): 浏览器打任意节点IP → 查 Leader → 转发
```

控制面 3 节点本身也兼作 Agent。强一致数据进 Raft 状态机(bbolt 后端), 瞬时状态(Agent 心跳、负载)存 Leader 内存。

### 数据存放策略

| 数据 | 存放 | 一致性 |
|------|------|--------|
| 服务器清单、服务定义 | Raft 状态机 | 强一致 |
| 配置基准版本、比对规则 | Raft 状态机 | 强一致 |
| 部署/迁移任务计划 | Raft 状态机 | 强一致 |
| Agent 实时心跳、负载、进程状态 | 内存 | 弱一致 |
| 操作审计日志 | 独立存储(后续) | 追加写 |

原则: 必须一致的进 Raft, 差不多就行的放内存。Raft 写有成本, 不滥用。

### 一次"修改 Java 配置"的完整流程

1. 浏览器打到任意节点, 入口代理查到 Leader 转发
2. Leader 生成配置变更记录, 写 Raft 日志, 复制到多数派, commit, apply 到状态机
3. Leader 调度器发现配置变了, 生成部署任务(也走 Raft), 下发目标 Agent
4. Agent 拉取新配置、写本地文件、重启 Java 进程、回报结果
5. Agent 回报 → Leader 更新实例状态(弱一致内存)

故障语义: Leader 挂在 commit 前 → 未确认, 客户端失败重试; 挂在 commit 后 → 多数派已有日志, 新 Leader 不丢。

## 五、技术栈

| 层 | 选型 | 备注 |
|----|------|------|
| 后端语言 | Go 1.26+ | 单二进制部署, Raft 生态最成熟 |
| Raft | hashicorp/raft v1.x | Consul/Nomad 同款 |
| 存储 | bbolt | Raft FSM 后端, 嵌入式 KV |
| Agent↔Leader | gRPC + protobuf | 双向流, keepalive |
| 前端 | Vue 3 + TypeScript + Vite | |
| UI 组件 | Element Plus | 表格/表单/树/抽屉 |
| 图表 | ECharts 5 | 资源监控曲线 |
| 拓扑可视化 | AntV G6 | 关系图/拓扑, 优于 ECharts graph |
| 配置编辑 | Monaco Editor | yml 编辑 + diff |
| 打包 | Go embed | 前端打进后端单二进制 |

后端选 Go 而非 Java 的理由: 单二进制部署对 bootstrap 场景(推 Agent 到 20 台)价值极大, Raft 生态 Go 最厚, 运维工具用 Go 是行业默认。代价是团队需学习 Go, 边做边学。

## 八、关键设计决策记录

1. **存储层一开始就用 Raft**: 即使 v0.1 单节点也用 Raft 库存数据, 避免"单机 DB 换 Raft"的大返工。这一步决策在 M1 落地, M2 升级 bbolt 时上层零改动验证了其价值。
2. **不用纯 mesh**: 强一致写场景 mesh 要叠 Raft, 复杂度高于直接用 Raft 小集群。
3. **Agent 在线状态不过 Raft**: 瞬时高频数据走内存, 不付一致性成本。
4. **前端 embed 进后端二进制**: bootstrap 推送单文件即可, 比前后端分离 + nginx 简单一个量级。
5. **gRPC 双向流而非 WebSocket**: 长连接 + 双向 + 强类型, 正匹配 Agent 通信模式。
