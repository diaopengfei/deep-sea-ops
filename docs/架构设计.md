# 架构设计

本文档是 deepsea-ops 项目的总体架构说明, 描述系统目标、拓扑选型、技术栈和演进路径。分布式一致性原理见 [Raft原理详解.md](./Raft原理详解.md), 代码层面见 [后端代码导读.md](./后端代码导读.md)。

---

## 一、项目目标

deepsea-ops 是一个分布式服务器运维平台, 用于管理维护 20+ 台服务器上的:

- Java 微服务 / Java 程序 / Python 程序
- Redis、PostgreSQL、Kafka、Elasticsearch、ClickHouse、Nacos

首期聚焦 Java 微服务管理, 包括:

- 配置文件维护修改: 连接 Nacos、Java 本地配置文件、jar 包内部配置文件进行读取比对
- 扩容与迁移: 对 Java 程序在不同服务器上进行扩容和迁移

整体要求:

- 分布式部署, 20+ 台服务器任意一台都可访问
- 节点间心跳连接
- 一台部署后导入服务器名单, 自动连接、自动部署本项目

本项目作为开源项目维护。

## 二、开源项目参考

调研过的同类项目及其对本项目的借鉴价值:

| 项目 | 架构特点 | 借鉴点 |
|------|---------|--------|
| Jpom (dromara) | Server + Agent 双端, Java 后端 + Vue 前端, Agent 长连 Server | 头号参考, 节点分发、项目管理模式 |
| Spug | Agentless SSH, Python(Django) + Vue | 轻量, 但不满足对等集群诉求 |
| 1Panel | Go + Vue, 单机面板 | UI 与资源监控设计 |

| Ansible / AWX | 无 Agent, playbook 幂等 | bootstrap 自动部署思路 |
| Consul / Nomad | Go, gossip 心跳 + Raft 调度 | 工业级对等集群参考 |
| Apollo / Nacos | 配置中心 | 客户端缓存 + 版本对比 |

业界两条主线: Agent 长连(Jpom 式)与 Agentless SSH(Spug 式)。本项目需求含"任意节点可访问、互相心跳、自动部署", 选 **Agent + 集群** 路线。

## 三、分布式拓扑选型

服务器间网络全互通。对比三种拓扑:

### 模式 A: Server + Agent 主从

- 一个 Server 集群接所有 Agent, Agent 单向长连
- 任意节点访问靠反代硬凑, 非真对等
- 实现最简, 但 Server 单点风险, "任意节点当入口"是伪命题

### 模式 B: 全对等 mesh

- 每台都跑相同节点, gossip 互相心跳
- 真任意节点入口, 无单点
- 强一致写(配置管理)需叠 Raft, 复杂度暴增, 20 台规模过度

### 模式 C: 混合(控制面小集群 + 全员 Agent)

- 3 节点 Raft 控制面 + 所有机器跑 Agent
- Raft 原生强一致, 秒级故障切换
- 入口代理让任意节点 IP 都能访问 UI
- 复杂度介于 A 与 B 之间, 正好匹配 20 台规模

### 选定: 模式 C

选 C 的三个理由:

1. "任意一台可访问、互相心跳" 是对等入口诉求, A 做不到真任意节点, B 代价大, C 用轻量入口代理精准命中
2. "配置文件比对修改" 是强一致写场景, Raft 比 mesh 最终一致靠谱
3. 20 台规模下 B 的 gossip 复杂度是负收益, C 的 Raft 一次性投入后续受益

## 四、模式 C 架构总览

```
┌─────────────────────────────────────────────────┐
│              控制面 (3 节点 Raft)                │
│   N1: Leader   N2: Follower   N3: Follower      │
│   每节点内: API + Raft 引擎 + 调度器            │
│   强一致状态: 服务器清单/服务定义/配置基准/部署计划 │
└─────────────────────────────────────────────────┘
        ▲心跳上报            ▲心跳上报
   ┌─────┴─────┐       ┌─────┴─────┐  ... 共 20 台
   │  Agent@N4 │       │  Agent@N5 │
   │ 执行部署/ │       │ 执行部署/ │
   │ 读配置/   │       │ 读配置/   │
   │ 起停进程  │       │ 起停进程  │
   └───────────┘       └───────────┘
   入口代理(每台都跑): 浏览器打任意节点IP → 查 Leader → 转发
```

控制面 3 节点本身也兼作 Agent。强一致数据进 Raft 状态机(bbolt 后端), 瞬时状态(Agent 心跳、负载)存 Leader 内存。

### 数据存放策略

| 数据 | 存放 | 一致性 |
|------|------|--------|
| 服务器清单、服务定义 | Raft 状态机 | 强一致 |
| 配置基准版本、比对规则 | Raft 状态机 | 强一致 |
| 部署/迁移任务计划 | Raft 状态机 | 强一致 |
| Agent 实时心跳、负载、进程状态 | 内存 | 弱一致 |
| 操作审计日志 | 独立存储(后续) | 追加写 |

原则: 必须一致的进 Raft, 差不多就行的放内存。Raft 写有成本, 不滥用。

### 一次"修改 Java 配置"的完整流程

1. 浏览器打到任意节点, 入口代理查到 Leader 转发
2. Leader 生成配置变更记录, 写 Raft 日志, 复制到多数派, commit, apply 到状态机
3. Leader 调度器发现配置变了, 生成部署任务(也走 Raft), 下发目标 Agent
4. Agent 拉取新配置、写本地文件、重启 Java 进程、回报结果
5. Agent 回报 → Leader 更新实例状态(弱一致内存)

故障语义: Leader 挂在 commit 前 → 未确认, 客户端失败重试; 挂在 commit 后 → 多数派已有日志, 新 Leader 不丢。

## 五、技术栈

| 层 | 选型 | 备注 |
|----|------|------|
| 后端语言 | Go 1.22+ | 单二进制部署, Raft 生态最成熟 |
| Raft | hashicorp/raft v1.x | Consul/Nomad 同款 |
| 存储 | bbolt | Raft FSM 后端, 嵌入式 KV |
| Agent↔Leader | gRPC + protobuf | 双向流, keepalive |
| 前端 | Vue 3 + TypeScript + Vite | |
| UI 组件 | Element Plus | 表格/表单/树/抽屉 |
| 图表 | ECharts 5 | 资源监控曲线 |
| 拓扑可视化 | AntV G6 | 关系图/拓扑, 优于 ECharts graph |
| 配置编辑 | Monaco Editor | yml 编辑 + diff |
| 打包 | Go embed | 前端打进后端单二进制 |

后端选 Go 而非 Java 的理由: 单二进制部署对 bootstrap 场景(推 Agent 到 20 台)价值极大, Raft 生态 Go 最厚, 运维工具用 Go 是行业默认。代价是团队需学习 Go, 边做边学。

## 六、目录结构

```
deepsea-ops/
├── docs/                       文档
├── scripts/                    启动 / 停止脚本 (start.sh/ps1, stop.sh/ps1)
├── config/                     配置文件示例 (server.yaml.example, agent.yaml.example)
├── server/                     Go 后端(module: github.com/deepsea-ops/server)
│   ├── go.mod / go.sum
│   ├── proto/agent.proto       gRPC 通信契约
│   ├── cmd/
│   │   ├── server/main.go      控制面入口 (-config 指定 YAML 配置)
│   │   └── agent/main.go       Agent 入口 (-config 指定 YAML 配置)
│   └── internal/               Go 强制私有包
│       ├── model/              领域模型(Server/User/Project/DeployTask/SSHCredential)
│       ├── store/              Raft 存储层(FSM/Store/Command, 5 个 bbolt bucket)
│       ├── api/                HTTP 路由 + 入口代理(Leader 转发)
│       ├── grpcserver/         Agent gRPC 连接管理
│       ├── agentclient/        Agent 端逻辑(连接/扫描/部署/进程检测)
│       ├── auth/               JWT + bcrypt + 登录限流
│       ├── crypto/             AES-GCM 加密(SSH 凭据)
│       ├── sshclient/          SSH 远程操作(连接/上传/命令)
│       ├── inject/             自动注入(SSH 推送 + systemd)
│       ├── config/             YAML 配置文件加载
│       ├── configdiff/         三路配置 diff
│       └── proto/agent/        protoc 生成代码
└── web/                        Vue 前端
    └── src/
        ├── api/                接口封装 + 共享类型
        ├── views/              页面组件
        ├── styles/             全局样式
        └── App.vue / main.ts
```

约定: `cmd/<名字>/` 每个子目录一个可执行程序; `internal/` 外部 module 不可 import, 语言级封装; 依赖方向 `main → api → store → model` 单向不循环。

## 七、演进路径

采用"分阶段、一次只啃一个硬骨头"策略。存储层从第一天就用 Raft, 避免后续返工。

### v0.1 单节点控制面 + Agent 骨架 ✅ 已完成

- M1 Raft 单节点存储层 ✅ (hashicorp/raft 单节点, FSM Apply/Snapshot/Restore)
- M2 bbolt 持久化 FSM ✅ (内存 map 换 bbolt, 接口不变)
- M3 gRPC 通信 + Agent 骨架 ✅ (proto 定义, 双向流, Agent 注册/心跳)
- M4 读 Java 配置回传 ✅ (控制面下发 READ_CONFIG → Agent 读文件 → 回传展示)

产出: 单节点系统能跑。前端可视化管理服务器、查看 Agent、远程读配置。

### v0.2 3 节点容错集群 ✅ 已完成

- 控制面 1 → 3 节点, 动态 AddVoter 加节点
- 命令行 flag 参数化(nodeID/raftAddr/joinAddr)
- 集群管理接口(/api/cluster/join, /api/cluster/info)
- 验证: 杀 Leader 后秒级选出新 Leader, 已提交数据不丢

产出: 真正的分布式容错。3 节点容忍 1 故障, 多数派继续服务。FSM/Apply/Snapshot 零改动, 验证了 v0.1 存储抽象的价值。

### v0.3 Java 运维 MVP + 安全鉴权 ✅ 已完成

- M1 登录鉴权 ✅
  - 登录页、JWT 签发/校验、API 鉴权中间件
  - bcrypt 密码加盐哈希、前端路由守卫
  - 登录失败限流防爆破、CORS 白名单
- M2 配置比对 ✅
  - Nacos / 本地配置文件 / jar 内配置 三源采集 + diff
  - 支持 Nacos 认证(accessToken 或 username/password 登录)
  - 前端 git 风格 diff 展示(行级颜色标注 + 来源徽章 + 过滤)
- M3 Agent 自动扫描 ✅
  - 可配置扫描目录(默认 /home、/data), 识别 Java Spring / Java jar / Python 项目
  - 读 Agent 节点 /etc/hosts 文件, 回传服务器间域名映射
  - 进程检测: 读 /proc/*/cmdline(Linux) 或 tasklist(Windows) 判断项目运行状态 + PID
  - 生效配置合并: 对 Spring 项目自动采集三路配置, 按 Spring 优先级(jar < 本地 < Nacos)合并, 记录覆盖关系
  - 前端项目扫描页: 项目列表(含运行状态) + 详情抽屉(生效配置 / 三路原始配置 / hosts)
- M4 配置自动发现与比对增强 ✅
  - 配置比对页改为"选 Agent → 选项目 → 自动填充 → 比对"流程, 复用 M3 扫描结果
  - 扫描结果走 Raft 持久化(projects bucket), 多节点共享视图
- M5 扩容迁移 ✅
  - 扩容: 选项目 + 选目标 Agent, 控制面下发部署指令(jar 分发、配置写入、进程启动)
  - 迁移: 旧节点停服 → 新节点起服, 编排走 Raft 保证一致性
  - 部署任务走 Raft 持久化(deploy_tasks bucket), 异步执行, 状态实时回传
- M6 拓扑可视化 ✅ (AntV G6 集成)
  - 服务器/服务/Agent 节点拓扑图, Leader/Follower/Agent 状态高亮
  - 集群概览(节点数/在线 Agent/Leader/Term), 自动刷新

### v0.4 自动部署 + 入口代理 ✅ 已完成

- 单机起步: 添加服务器 SSH 连接信息(AES-GCM 加密存 Raft)
- 角色选择 UI: 勾选 Raft 节点(校验奇数≥3)/ Agent 节点
- 自动注入: SSH 推送二进制 + 配置, 远程拉起 systemd, Raft 节点自动 join, Agent 自动连 Leader
- 入口代理: 任意节点 IP 可访问 UI, 自动转发当前 Leader(写请求转发, 读请求本地)
- 凭据加密: SSH 私钥/密码 AES-GCM 加密, 主密钥从环境变量 MASTER_KEY
- 深度代码审查: 修复 17 处缺陷 (静默吞错、逻辑 bug、OOM 风险、死代码)

产出: 一台部署、导入名单、自动连自动部署的完整体验。

### v0.5 配置文件启动 + 服务器管理重构 ✅ 已完成

- YAML 配置文件启动: 参考 Kafka / Elasticsearch, `-config` 指定路径, 默认 `config/server.yaml` / `config/agent.yaml`, 文件不存在时回退内置默认值(向后兼容)
- 服务器模型重构: ID 由字符串改为数字自增(FSM 分配, Raft Apply 顺序执行保证并发安全), 新增 OS 类型字段(Linux / Windows, 默认 Linux)
- SSH 凭据内置: 服务器导入时填入 SSH 用户名/密码, 密码 AES-GCM 加密存储, 支持连接测试
- 服务器列表查询: 支持全字段排序(sort + order 参数)和全字段模糊检索(keyword 参数), 服务端处理
- 服务器增删改: 新增 UpdServer / DelServer Raft 命令, 支持编辑和删除
- 清理无用文档和脚本: 删除 Agent 交接文档, 启动脚本改为生成配置文件启动

产出: 工业级配置文件启动方式 + 完整的服务器管理 CRUD。

### v0.5.1 安全配置纳入配置文件 ✅ 已完成

- `server.yaml` 新增 `security` 段: `jwt_secret` / `admin_password` / `master_key`
- 配置优先级: 环境变量 > YAML 配置 > 内置默认值 (容器化部署用环境变量, 物理机部署用配置文件)
- 多节点 Raft 集群一致性: `jwt_secret` 和 `master_key` 必须所有节点一致 (入口代理转发 + 凭据解密需要)
- 启动时安全配置校验: 默认值警告 / 未设置警告 / join 模式强警告
- `crypto` 和 `auth` 包改为显式初始化 (InitMasterKey / InitJWTSecret), 消除隐式环境变量依赖
- 解决问题: 原 MASTER_KEY 开发模式随机生成, 多节点下 Follower 当选 Leader 后无法解密 Raft 中的凭据

产出: 安全配置统一管理, 多节点部署不再因密钥不一致而崩溃。

## 八、关键设计决策记录

1. **存储层一开始就用 Raft**: 即使 v0.1 单节点也用 Raft 库存数据, 避免"单机 DB 换 Raft"的大返工。这一步决策在 M1 落地, M2 升级 bbolt 时上层零改动验证了其价值。
2. **不用纯 mesh**: 强一致写场景 mesh 要叠 Raft, 复杂度高于直接用 Raft 小集群。
3. **Agent 在线状态不过 Raft**: 瞬时高频数据走内存, 不付一致性成本。
4. **前端 embed 进后端二进制**: bootstrap 推送单文件即可, 比前后端分离 + nginx 简单一个量级。
5. **gRPC 双向流而非 WebSocket**: 长连接 + 双向 + 强类型, 正匹配 Agent 通信模式。
