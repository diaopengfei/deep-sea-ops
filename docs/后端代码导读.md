# 后端代码导读(零 Go 基础版)

这份文档帮你读懂 `server/` 下的全部后端代码。先讲 Go 语言里反复出现的语法,再逐文件解读,最后讲清楚依赖和数据流。

---

## 第一部分:Go 语法速查(只讲本项目用到的)

读代码前,先记住这几条 Go 规则,它们和 Java/JS 都不太一样。

### 1. package 与 import

```go
package main        // 声明这个文件属于哪个包
import (            // 导入依赖, 类似 Java 的 import
    "log"
    "net/http"
    "github.com/deepsea-ops/server/internal/store"
)
```

- 每个 `.go` 文件第一行必须声明 `package`。同一个目录下的所有文件属于同一个包。
- `package main` 是特殊的:它表示这是一个**可执行程序**(有 `main()` 入口),其他包都是库。
- `import` 的路径里带域名的(如 `github.com/...`)是**第三方/外部包**,需要 `go mod` 管理;不带域名的(如 `log`)是**标准库**,直接用。

### 2. 可见性:首字母大小写决定"公开/私有"

这是 Go 最反直觉的规则,**没有 `public`/`private` 关键字**:

- 首字母**大写** = 包外可访问(导出),相当于 Java 的 `public`。
- 首字母**小写** = 仅包内可访问,相当于 Java 的 `private`。

举例:`func New(...)` 别的包能调用;`func handleListServers(...)` 只能在 `api` 包内部用。结构体字段同理:`Server.ID`(大写)外部可见,`fsmSnapshot.data`(小写)外部不可见。

### 3. 变量声明与短变量声明 `:=`

```go
var cmd command           // 声明变量 cmd, 类型 command, 零值
fsm := NewFSM()           // 短变量声明: 自动推断类型, 最常用
err := json.Unmarshal(...) // 多返回值, 见下条
```

- `:=` 是"短变量声明",自动推断类型,函数内部最常用。
- `var` 用于需要显式指定类型或声明包级变量时。

### 4. 多返回值与 error

```go
func NewStore(...) (*Store, error) { ... }
storeInstance, err := store.NewStore(...)
if err != nil {
    log.Fatalf("启动失败: %v", err)
}
```

- Go 函数可以返回多个值,惯例是**最后一个返回 `error`**。
- **没有 try/catch**。错误必须逐个检查:`if err != nil { ... }`。这是 Go 代码看起来"啰嗦"的根本原因,但好处是错误处理路径很显式。
- `nil` 是空值(类似 Java 的 `null`),但 error 是接口类型,`nil` 表示"没有错误"。

### 5. 结构体 struct(类似 Java 的类)

```go
type Server struct {
    ID     string `json:"id"`     // 字段 类型 标签
    Name   string `json:"name"`
}
```

- `type Server struct {...}` 定义结构体,相当于 Java 的 class(但 Go 没有"类"的概念,只有结构体+方法)。
- 字段后面反引号里的 `` `json:"id"` `` 叫**结构体标签(struct tag)**,是给反射用的元数据。这里告诉 JSON 编码器:序列化时这个字段用 `"id"` 作为 key。
- 创建实例:`Server{ID: "s1", Name: "web"}` 或 `&Server{...}`(取指针)。

### 6. 方法(method):给结构体绑函数

```go
func (f *FSM) Apply(l *raft.Log) interface{} { ... }
//     ^^^^   接收者, 类似 Java 的 this
```

- `(f *FSM)` 是**接收者(receiver)**,表示这个函数属于 `FSM` 类型,`f` 相当于 Java 的 `this`。
- `*FSM` 中的 `*` 表示指针(传引用,能改原对象);不带 `*` 是值拷贝。

### 7. 接口(interface):隐式实现

```go
// raft 库定义的接口(简化)
type FSM interface {
    Apply(*Log) interface{}
    Snapshot() (FSMSnapshot, error)
    Restore(io.ReadCloser) error
}
```

- Go 的接口是**隐式**的:你只要实现了接口里所有方法,就算实现了这个接口,**不需要 `implements` 关键字**。
- 我们的 `FSM` 结构体实现了 `Apply`/`Snapshot`/`Restore` 三个方法,就自动满足 `raft.FSM` 接口,可以传给 Raft。这叫"鸭子类型"。

### 8. defer:函数返回前执行

```go
defer f.mu.Unlock()
defer rc.Close()
```

- `defer` 语句在**所在函数返回时**执行,常用于释放锁、关闭文件,类似 Java 的 `finally`。
- 多个 defer 按 LIFO(后进先出)顺序执行。

### 9. goroutine 与并发(本项目暂未显式用,但 Raft 内部用)

```go
go someFunction()   // 启动一个 goroutine(轻量线程)
```

- `go 函数()` 开一个 goroutine 并发执行。本项目里 Raft 库内部大量使用,我们自己的代码暂时没显式开。
- `sync.RWMutex`(读写锁)用来保护并发访问的 map,见 `fsm.go`。

### 10. slice 与 map

```go
out := make([]model.Server, 0, len(f.servers))  // slice, 类似动态数组
servers := make(map[string]model.Server)          // map, 类似 HashMap
f.servers[cmd.Server.ID] = cmd.Server             // 写 map
```

- `[]T` 是 slice(切片),动态数组,类似 Java 的 `ArrayList`。
- `map[K]V` 是哈希表,类似 Java 的 `HashMap`。
- `make(...)` 用于初始化 slice/map/channel。

## 第二部分:项目结构与依赖关系

```
server/
├── go.mod                       模块定义 + 依赖清单
├── cmd/server/main.go           程序入口 (package main)
└── internal/                    Go 强制的私有包边界
    ├── model/                  领域模型 (package model)
    ├── store/                  Raft 存储层 (package store)
    │   ├── command.go          Raft 命令定义
    │   ├── fsm.go              状态机实现
    │   └── store.go            Raft 封装 + 对外读写 API
    └── api/                    HTTP 路由 + handler (package api)
        ├── router.go           路由注册
        ├── server_handler.go   服务器 CRUD
        └── ...                 按领域拆分的 handler 文件
```

> 本文档聚焦 v0.1 核心概念(Raft + Store + API 最小闭环)。项目后续版本新增了 `grpcserver`/`agentclient`/`platform`/`scheduler`/`auth`/`crypto`/`sshclient`/`inject`/`config`/`configdiff`/`shellutil` 等包, 完整结构见 [架构设计.md](./架构设计.md)。

**依赖方向(单向, 不会循环)**:

```
main ──> api ──> store ──> model
                   │
                   └──> (hashicorp/raft, raft-boltdb)  外部依赖
```

- `model` 在最底层,不依赖任何项目内包。
- `store` 依赖 `model` 和外部 Raft 库。
- `api` 依赖 `store` 和 `model`。
- `main` 组装 `store` 和 `api`。

**`internal/` 的特殊含义**:Go 编译器规定,`internal/` 下的包**只能被同一 module 内的代码 import**。外部项目想引用 `internal/store` 会编译失败。这是语言层面的"私有实现"声明。

## 第三部分:逐文件解读

### 1. `go.mod` —— 模块与依赖清单

```
module github.com/deepsea-ops/server   模块路径(唯一标识)
go 1.26.4                               最低 Go 版本
require (...)                           直接依赖
require (...) // indirect               间接依赖(直接依赖的依赖)
```

- 类似 Java 的 `pom.xml` 或 Node 的 `package.json`。
- `github.com/hashicorp/raft` 和 `raft-boltdb/v2` 是我们直接 import 的。
- `// indirect` 的那些是 Raft 自己依赖的库,Go 自动分析出来的,你不用管。
- `go.sum` 记录每个依赖的哈希值,保证下载的依赖没被篡改(类似 `package-lock.json`)。

### 2. `internal/model/server.go` —— 领域模型

```go
package model

type Server struct {
    ID     string `json:"id"`
    Name   string `json:"name"`
    IP     string `json:"ip"`
    Status string `json:"status"`
}
```

整个文件就定义了一个 `Server` 结构体。它对应前端 `api/types.ts` 里的 `Server` 接口,两边字段一一对应。`json` 标签保证序列化成 JSON 时 key 是小写的 `id`/`name` 等(否则 Go 默认用字段名 `ID`/`Name`)。

### 3. `internal/store/command.go` —— Raft 命令定义

```go
type command struct {
    Op     string       `json:"op"`      // "add"
    Server model.Server `json:"server"`
}
```

**关键概念**:Raft 不认识我们的业务结构,它只负责按顺序复制"一段字节"。所以我们把要做的事(比如"新增一台服务器")打包成 `command`,序列化成 JSON 字节交给 Raft。等 Raft 确认后,它会把同样的字节回放给 `FSM.Apply`。

`command` 是小写开头,只在 `store` 包内用,外部看不到——这是私有实现细节。

### 4. `internal/store/fsm.go` —— 状态机(本项目的核心)

这是 Raft 留给业务代码的接口实现。Raft 负责"可靠地按顺序送达命令",FSM 负责"收到命令后怎么改状态"。FSM 必须实现三个方法:`Apply` / `Snapshot` / `Restore`。

#### 结构体与构造

```go
type FSM struct {
    mu      sync.RWMutex            // 读写锁, 保护并发访问
    servers map[string]model.Server // 内存存储, key=Server.ID
}

func NewFSM() *FSM {
    return &FSM{servers: make(map[string]model.Server)}
}
```

- `mu sync.RWMutex` 是读写锁。`RLock()` 读锁(可多读并发),`Lock()` 写锁(独占)。因为 `Apply`(写)和 `List`(读)可能并发,必须加锁。
- `servers` 是内存 map。M2 会换成 bbolt 持久化,但这个文件的结构和接口签名不变。

#### Apply —— 收到命令时改状态

```go
func (f *FSM) Apply(l *raft.Log) interface{} {
    var cmd command
    if err := json.Unmarshal(l.Data, &cmd); err != nil {
        return fmt.Errorf("反序列化命令失败: %w", err)
    }
    f.mu.Lock()
    defer f.mu.Unlock()
    switch cmd.Op {
    case "add":
        f.servers[cmd.Server.ID] = cmd.Server
        return nil
    default:
        return fmt.Errorf("未知操作: %s", cmd.Op)
    }
}
```

- `l *raft.Log`:`l.Data` 就是我们当初传给 `raft.Apply` 的字节。
- `json.Unmarshal(l.Data, &cmd)`:`&cmd` 取地址,把字节反序列化进 cmd。
- 返回值会传回给 `raft.Apply` 的调用方,我们用它返回错误(`nil` 表示成功)。
- **注意参数名用 `l` 而不是 `log`**:否则会遮蔽 `log` 包,导致 `log.Printf` 编译失败。这是 Go 作用域的坑。

#### List —— 读当前状态(不过 Raft)

```go
func (f *FSM) List() []model.Server {
    f.mu.RLock()
    defer f.mu.RUnlock()
    out := make([]model.Server, 0, len(f.servers))
    for _, s := range f.servers {
        out = append(out, s)
    }
    return out
}
```

- 读操作直接读内存,不走 Raft 一致性流程。这是"脏读"优化——读多写少的场景下读很快。
- `for _, s := range f.servers`:`_` 是空白标识符,丢弃 key(我们只要 value `s`)。
- `make([]T, 0, n)`:预分配容量 n 的空 slice,减少扩容。

#### Snapshot / Restore —— 快照与恢复

```go
func (f *FSM) Snapshot() (raft.FSMSnapshot, error) {
    // 深拷贝当前 map, 避免快照过程中被 Apply 改动
    clone := make(map[string]model.Server, len(f.servers))
    for k, v := range f.servers { clone[k] = v }
    return &fsmSnapshot{data: snapshotData{Servers: clone}}, nil
}

func (f *FSM) Restore(rc io.ReadCloser) error {
    defer rc.Close()
    var data snapshotData
    gob.NewDecoder(rc).Decode(&data)
    f.servers = data.Servers
    return nil
}
```

- **Snapshot**:Raft 积累一定量日志后会调它,把当前状态打包,用来压缩日志(旧日志可以丢弃)。
- **Restore**:节点启动时先从快照恢复,再重放快照之后的日志。
- `gob` 是 Go 自带的二进制序列化格式,比 JSON 紧凑,用于快照。
- `fsmSnapshot.Persist` 把快照写到 Raft 提供的 `sink`;`Release` 是清理钩子(这里空实现)。

### 5. `internal/store/store.go` —— Raft 封装

这个文件负责把 Raft 实例组装起来,对外暴露简单的 `AddServer` / `ListServers`。上层 API 只跟 `Store` 打交道,不直接碰 `raft.Raft` 的细节。

#### Store 结构

```go
type Store struct {
    raft *raft.Raft
    fsm  *FSM
}
```

#### NewStore —— 启动 Raft 单节点

这是最长的函数,但它做的事是固定的"组装流水线":

```go
func NewStore(raftDir, raftAddr string) (*Store, error) {
    // 1. 准备数据目录
    os.MkdirAll(raftDir, 0o755)

    // 2. 创建 FSM
    fsm := NewFSM()

    // 3. Raft 配置
    config := raft.DefaultConfig()
    config.LocalID = "node1"                    // 节点 ID
    config.SnapshotInterval = 30 * time.Second   // 多久做一次快照
    config.SnapshotThreshold = 2                 // 积累 2 条日志就可能快照

    // 4. 日志存储(bbolt): Raft 命令日志持久化在这
    logStore, _ := raftboltdb.NewBoltStore(filepath.Join(raftDir, "raft.db"))

    // 5. 快照存储(文件系统)
    snapshotStore, _ := raft.NewFileSnapshotStore(raftDir, 1, os.Stderr)

    // 6. 传输层(TCP): 节点间通信, 复制日志/投票
    transport, _ := raft.NewTCPTransport(raftAddr, nil, 3, 10*time.Second, os.Stderr)

    // 7. 组装 Raft 实例
    r, _ := raft.NewRaft(config, fsm, logStore, logStore, snapshotStore, transport)

    // 8. 引导单节点集群(首次启动)
    r.BootstrapCluster(raft.Configuration{
        Servers: []raft.Server{{ID: config.LocalID, Address: transport.LocalAddr()}},
    })

    // 9. 等待自己成为 Leader
    s.waitForLeader(10 * time.Second)
}
```

几个要点:
- `0o755` 是八进制文件权限(Unix 风格,Windows 上部分生效)。
- `logStore` 同时用作"日志存储"和"稳定存储"两个参数——Raft 要求这两个接口,我们用同一个 bbolt 文件实现。
- `BootstrapCluster` 只在首次启动需要,重启时已引导过会返回 `ErrCantBootstrap`,代码里忽略这个特定错误。
- `errors.Is(err, raft.ErrCantBootstrap)` 判断错误类型,类似 Java 的 `instanceof`。

#### waitForLeader —— 轮询等当选

```go
func (s *Store) waitForLeader(timeout time.Duration) error {
    deadline := time.Now().Add(timeout)
    for time.Now().Before(deadline) {
        if s.raft.State() == raft.Leader { return nil }
        time.Sleep(100 * time.Millisecond)
    }
    return errors.New("等待 Leader 超时")
}
```

Raft 写操作必须经过 Leader,启动后要先等它当选。单节点几秒内就会当选。

#### AddServer —— 提交写命令(过 Raft 一致性)

```go
func (s *Store) AddServer(srv model.Server) error {
    cmd := command{Op: "add", Server: srv}
    data, _ := json.Marshal(cmd)                 // 命令 → JSON 字节
    f := s.raft.Apply(data, 5*time.Second)       // 提交给 Raft, 5 秒超时
    if err := f.Error(); err != nil { return err }
    if resp := f.Response(); resp != nil { ... } // FSM.Apply 的返回值
    return nil
}
```

这就是"写操作过 Raft"的完整路径:`序列化 → raft.Apply → 复制+commit → FSM.Apply → 返回`。

`raft.Apply` 返回一个 **future**(异步结果对象),`f.Error()` 阻塞等待命令完成。这种"future + 显式等待"是 Go 里处理异步的常见模式(不用 async/await)。

#### ListServers —— 读(不过 Raft)

```go
func (s *Store) ListServers() []model.Server {
    return s.fsm.List()
}
```

直接委托给 FSM 的内存读。

### 6. `internal/api/` —— HTTP 路由

> v0.6.1 起, 原 `api/server.go`(1159 行)按领域拆分为 8 个文件: `router.go`(路由注册)、`server_handler.go`(服务器 CRUD)、`agent_handler.go`、`deploy_handler.go`、`inject_handler.go`、`opsnode_handler.go`、`credential_handler.go`、`proxy.go`(Leader 转发)。下面以 `router.go` 中的路由注册为例。

#### New —— 构造路由(导出函数,大写 N)

```go
func New(s *store.Store) http.Handler {
    mux := http.NewServeMux()
    mux.HandleFunc("/api/servers", func(w http.ResponseWriter, r *http.Request) {
        switch r.Method {
        case http.MethodGet:  handleListServers(w, s)
        case http.MethodPost: handleAddServer(w, r, s)
        default: w.WriteHeader(http.StatusMethodNotAllowed)
        }
    })
    return mux
}
```

- `func(w, r)` 是**匿名函数(闭包)**,捕获了外层的 `s`。这是 Go 的轻量依赖注入——不用框架,直接闭包传依赖。
- `switch r.Method` 按 HTTP 方法分发。Go 的 `switch` **默认不穿透**(不像 C/Java 要写 break),匹配一个 case 就退出。
- 返回 `http.Handler` 接口,任何实现了 `ServeHTTP` 的类型都满足。

#### handleAddServer —— 处理 POST

```go
func handleAddServer(w http.ResponseWriter, r *http.Request, s *store.Store) {
    var srv model.Server
    if err := json.NewDecoder(r.Body).Decode(&srv); err != nil { ... }
    // 校验 + 默认值
    if err := s.AddServer(srv); err != nil { ... }   // 走 Raft 写入
    w.WriteHeader(http.StatusCreated)
    json.NewEncoder(w).Encode(srv)
}
```

- `json.NewDecoder(r.Body).Decode(&srv)`:流式解析请求体 JSON 进结构体。
- `http.StatusBadRequest`(400)、`http.StatusCreated`(201)等是 HTTP 状态码常量。

### 7. `cmd/server/main.go` —— 入口

```go
package main

func main() {
    storeInstance, err := store.NewStore("raft-data", "127.0.0.1:7000")
    if err != nil { log.Fatalf("启动 Raft 失败: %v", err) }

    handler := api.New(storeInstance)
    log.Fatal(http.ListenAndServe(":8080", handler))
}
```

- `package main` + `func main()` 是可执行程序的入口(类似 Java 的 `public static void main`)。
- `log.Fatalf`:打印后**直接退出进程**(调用 `os.Exit(1)`),用于不可恢复的启动错误。
- `log.Fatal(http.ListenAndServe(...))`:服务正常跑时这行会阻塞;返回说明出错了,用 `log.Fatal` 退出。
- `"raft-data"` 是相对路径,基于进程工作目录,生产环境应从配置文件读取。

## 第四部分:数据流(一次"新增服务器"的完整旅程)

```
浏览器(填表点新增)
    │ POST /api/servers  body: {"id":"s4","name":"web","ip":"...","status":"online"}
    ▼
Vite 代理(开发期) → http://localhost:8080/api/servers
    ▼
api.handleAddServer
    │ json.Decode → srv
    │ 校验 id/name/ip 非空
    ▼
store.AddServer(srv)
    │ command{Op:"add", Server:srv} → json.Marshal → 字节
    ▼
raft.Apply(字节, 5s)
    │ Raft 内部: 写日志(bbolt) → 复制(单节点跳过) → commit(多数派=自己)
    ▼
FSM.Apply(l)   ← Raft 回调, l.Data 就是上面的字节
    │ json.Unmarshal → cmd
    │ f.mu.Lock() → f.servers[cmd.Server.ID] = cmd.Server → defer Unlock
    │ return nil
    ▼
回到 raft.Apply 返回(future.Error() = nil)
    ▼
回到 store.AddServer → 返回 nil
    ▼
回到 api.handleAddServer
    │ w.WriteHeader(201) + json.Encode(srv)
    ▼
浏览器收到 201 + 新增的服务器 JSON
```

**读流程简单得多**:`浏览器 → api.handleListServers → store.ListServers → fsm.List()(直接读内存 map)→ JSON 返回`。读不过 Raft,所以很快。

## 第五部分:外部依赖说明

| 依赖 | 作用 | 类比 |
|------|------|------|

## 第六部分:常见疑问

**Q: 为什么 `Apply` 的返回值是 `interface{}`?**
`interface{}` 是空接口,任何类型都满足(类似 Java 的 `Object`)。Raft 不知道业务要返回什么,就用空接口兜底。我们在里面返回 `nil`(成功)或 `error`(失败)。

**Q: `:=` 和 `=` 有什么区别?**
`:=` 是声明并赋值(创建新变量);`=` 是纯赋值(变量已存在)。`x := 1` 声明 x;之后 `x = 2` 只是改值。

**Q: 为什么到处是 `if err != nil`?**
Go 没有 try/catch,错误是普通返回值,必须逐个检查。这看起来啰嗦,但错误处理路径完全显式,没有"隐藏的异常"。

**Q: `defer` 在哪执行?**
在**所在函数返回时**,按 LIFO 顺序。常用于 `defer mu.Unlock()`,保证函数无论怎么返回都会解锁。

**Q: 读为什么不过 Raft?**
Raft 保证的是**写**的一致性(所有节点按相同顺序应用相同命令)。读当前节点的内存是"脏读"——可能读到稍旧的数据,但单节点场景下没有"旧"的问题。多节点时如果要求强一致读,要经过 Leader 的 read index,这是 v0.2 之后的话题。

**Q: `internal/` 必须写吗?**
不写也能跑,但写了之后 Go 编译器会阻止外部项目 import 这些包。这是语言级别的封装,强烈推荐。Kubernetes、etcd 等大型项目都用这个约定。

## 阅读建议

1. 先读 `model/server.go`(最简单,一个结构体)。
2. 再读 `cmd/server/main.go`(看组装关系,建立全局印象)。
3. 然后读 `store/fsm.go`(核心,理解 Apply/Snapshot/Restore)。
4. 接着读 `store/store.go`(理解 Raft 怎么启动、AddServer 怎么走)。
5. 最后读 `api/router.go`(理解 HTTP 路由怎么注册), 再按需看对应领域的 `*_handler.go`。
6. `store/command.go` 随时翻,它只是一个数据结构定义。

读到不认识的语法,回到本文档第一部分查。带着"一次新增请求怎么走完整条链路"的问题去读第四部分,把每个文件对应到链路的某个环节,理解会深很多。
